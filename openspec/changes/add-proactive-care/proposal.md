# Proposal: 主动关怀与对话发起机制

## 概述
当前 Health Pilot 主要是一个“被动响应”的工具，用户输入数据，AI 记录并反馈。虽然已经具备了定时提醒和条件触发（如热量超标预警）的基础代码，但这些功能尚未与主干对话 Agent (Coach Agent) 形成闭环，缺乏真正的“主动关怀”体验。

本提案旨在实现四个核心场景：打通推送记忆闭环、实时事件触发、沉默用户唤醒以及推送消息的端侧聊天流渲染，从而让 AI 教练具备真正的主动性和人情味。

## 动机与目标
为了让 AI 教练从单纯的工具升级为有温度的健康伙伴，我们需要它能：
1. **“记得”自己说过的话**：当系统发了推送，用户回复时，AI 应该顺着语境聊，而不是一脸懵。
2. **“及时”给出反馈**：用户记录了一顿高热量晚餐后，系统应立即感知并主动提醒，而不是等用户问。
3. **“主动”关心用户**：在用户长时间未记录数据时，主动发消息询问近况，提升用户留存。

## 范围界定
### 包含 (In Scope)
- **打通记忆闭环**：在 `agents/coach.py` 中获取最近发送且未读的 Notification，注入到系统提示词中，让 Coach 知道推送上下文。
- **实时条件触发 (Event-Driven)**：在 `tools/meal_tracker.py` 和其他 Tracker 中，数据保存成功后异步或实时调用 `check_conditional_triggers`。
- **沉默唤醒 (Chaser)**：在 `scheduler/push_scheduler.py` 增加定时任务，每日检查无记录用户并生成关怀推送。
- **模拟端侧展示**：通过 API 改造，使得前端在拉取未读消息时，能将其视为 AI 教练主动发出的消息渲染在对话流中。

### 不包含 (Out of Scope)
- 接入真实的第三方推送通道（如微信、企业微信），目前仍以数据库 Notification 表和轮询 API 为主。
- 复杂的对话打断策略（如用户正在长篇大论输入时强行推送）。

## 成功指标
1. 用户在收到条件推送后回复，Coach 能够准确理解语境并继续对话。
2. 记录一顿热量超过阈值的饮食后，数据库中立即生成对应的条件推送记录。
3. 连续 24 小时无记录的用户，数据库中生成 `silent_wakeup` 类型的推送。

## 替代方案
- **维持现状**：仅依赖用户主动提问。这会导致交互单一，无法发挥健康管理“防患于未然”的价值。
- **独立多 Agent 互斥**：让 Push Agent 直接接管对话。这会导致上下文割裂，且实现复杂度高，不如直接在 prompt 层面注入推送上下文。
